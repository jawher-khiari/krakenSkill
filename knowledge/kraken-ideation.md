# Kraken Idea Refinement Reference

> Absorbed from [addyosmani/agent-skills](https://github.com/addyosmani/agent-skills) — idea-refine skill.

## Philosophy
- Simplicity is the ultimate sophistication
- Start with user experience, work backwards to technology
- Say no to 1,000 things — focus beats breadth
- Challenge every assumption — "how it's usually done" is not a reason
- The parts you can't see should be as beautiful as the parts you can

## Three-Phase Ideation

### Phase 1: Understand & Expand (Divergent)
1. Restate as a crisp "How Might We" problem statement
2. Ask 3-5 sharpening questions (no more):
   - Who is this for, specifically?
   - What does success look like?
   - What are the real constraints?
   - What's been tried before?
   - Why now?
3. Generate 5-8 idea variations using lenses:
   - **Inversion:** "What if we did the opposite?"
   - **Constraint removal:** "What if budget/time/tech weren't factors?"
   - **Audience shift:** "What if this were for [different user]?"
   - **Combination:** "What if we merged this with [adjacent idea]?"
   - **Simplification:** "What's the version that's 10x simpler?"
   - **10x version:** "What would this look like at massive scale?"
   - **Expert lens:** "What would domain experts find obvious?"

### Phase 2: Evaluate & Converge
1. Cluster resonating ideas into 2-3 distinct directions
2. Stress-test each against:
   - **User value:** Painkiller or vitamin?
   - **Feasibility:** Technical/resource cost? Hardest part?
   - **Differentiation:** Genuinely different? Would someone switch?
3. Surface hidden assumptions:
   - What you're betting is true (unvalidated)
   - What could kill this idea
   - What you're choosing to ignore (and why that's okay for now)

### Phase 3: Sharpen & Ship
Produce a one-pager:
- Problem Statement (How Might We)
- Recommended Direction (2-3 paragraphs max)
- Key Assumptions to Validate (with how to test each)
- MVP Scope (minimum version testing core assumption)
- **Not Doing list** (the most valuable part — explicit trade-offs)
- Open Questions

## Anti-Patterns
- Generating 20+ shallow variations instead of 5-8 considered ones
- Being a yes-machine instead of pushing back on weak ideas
- Skipping "who is this for"
- No assumptions surfaced before committing
- Producing a plan without a "Not Doing" list
- Jumping to output without running exploration first
