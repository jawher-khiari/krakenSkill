# Project Indexer

Creates and maintains `.kraken/INDEX.md` — a machine-readable codebase map that the agent reads every interaction to understand the project without re-scanning files.

**This is NOT documentation for humans. It is a structured reference for the agent.**

---

## Trigger Commands

| User says | Action |
|---|---|
| "init project" / "index" / "scan" | Full scan → create INDEX.md |
| "update index" | Re-scan changed files only (git diff) |
| (any file created/edited/deleted during Kraken P6) | Auto-update affected entries |
| (Kraken Phase 2 starts) | Read INDEX.md if exists, full scan if not |

---

## RULE: Read Before Write

**Every message that involves code changes:**
1. Read `.kraken/INDEX.md` first (if exists)
2. Use it to understand file purposes, exports, dependencies
3. After ANY file change, update the affected entries in INDEX.md
4. Never let INDEX.md go stale — it IS the agent's memory of the codebase

---

## Phase 1: DETECT — Identify Stack

Scan project root for framework indicators:

```
🔍 package.json     → Node.js (check for express, react, angular, next, nest)
🔍 *.csproj / *.sln → .NET (check target framework, packages)
🔍 requirements.txt / pyproject.toml → Python (check for django, flask, fastapi)
🔍 go.mod           → Go
🔍 Cargo.toml       → Rust
🔍 docker-compose*  → Container orchestration
🔍 .env / .env.*    → Environment config
🔍 tsconfig.json    → TypeScript
🔍 angular.json     → Angular
```

Output: Stack signature → `{backend}+{frontend}+{db}+{infra}`

---

## Phase 2: SCAN — Build Index

### File Classification Rules

For EVERY source file (not node_modules, not .git, not build artifacts):

```
FILE: [relative path]
TYPE: [controller|service|model|route|middleware|component|page|hook|util|config|test|migration|seed|script|style|asset]
PURPOSE: [1-line: what this file does — be specific]
EXPORTS: [named exports, class names, function names]
IMPORTS: [key internal dependencies — other project files this depends on]
DEPENDS-ON: [external packages used]
ROUTES: [HTTP routes defined here, if any: GET /api/users, POST /api/auth/login]
DB: [tables/collections accessed, if any]
LINES: [line count]
COMPLEXITY: [LOW|MEDIUM|HIGH — based on cyclomatic complexity estimate]
```

### Skip List (never index)
- `node_modules/`, `.git/`, `dist/`, `build/`, `bin/`, `obj/`
- `*.min.js`, `*.map`, `*.lock` (but DO index `package.json`, `*.csproj`)
- Binary files, images, fonts
- `.env` files (note existence but NEVER index contents)

### Grouping Strategy

Group files by architectural layer, not by directory:

```markdown
## API Layer
[controllers, routes, endpoints]

## Business Logic
[services, use-cases, domain logic]

## Data Layer
[models, repositories, migrations, seeds]

## Presentation Layer
[pages, components, layouts, hooks]

## Infrastructure
[config, middleware, auth, logging, Docker, CI]

## Tests
[unit, integration, e2e — grouped by what they test]
```

## Phase 3: GENERATE — Write INDEX.md

Create `.kraken/INDEX.md` with this exact structure:

```markdown
# PROJECT INDEX
<!-- Auto-generated by Kraken Project Indexer. Do NOT edit manually. -->
<!-- Last scan: {ISO timestamp} -->
<!-- Stack: {detected stack signature} -->
<!-- Files indexed: {count} -->

## STACK
- Backend: {framework + version}
- Frontend: {framework + version}
- Database: {type + ORM/driver}
- Runtime: {node version / dotnet version / python version}
- Package manager: {npm/yarn/pnpm/pip/dotnet}

## ARCHITECTURE PATTERN
{monolith | layered | clean-architecture | vertical-slice | microservices}
{brief 1-line description of how code is organized}

## ENTRY POINTS
- Server: {main entry file}
- Frontend: {main entry file}
- CLI: {if any}

## API LAYER
### {filename}
- **Path:** `{relative path}`
- **Purpose:** {what it does}
- **Routes:** {verb} {path} → {handler}
- **Auth:** {auth middleware if any}

## BUSINESS LOGIC
### {filename}
- **Path:** `{relative path}`
- **Purpose:** {what it does}
- **Exports:** {function/class names}
- **Depends on:** {other project files}

## DATA LAYER
### {filename}
- **Path:** `{relative path}`
- **Purpose:** {what it does}
- **Tables/Collections:** {names}
- **ORM:** {model relationships if detected}

## PRESENTATION LAYER
### {filename}
- **Path:** `{relative path}`
- **Purpose:** {what it does}
- **Props/Inputs:** {key props or @Input}
- **State:** {state management used}
- **Routes to:** {frontend route if page}

## INFRASTRUCTURE
### {filename}
- **Path:** `{relative path}`
- **Purpose:** {what it does}

## TESTS
### {filename}
- **Path:** `{relative path}`
- **Tests:** {what it tests}
- **Type:** {unit|integration|e2e}

## DEPENDENCY MAP
{adjacency list: file → [files it imports from this project]}

## ENV VARS
{list variable NAMES only from .env.example or docker-compose — NEVER values}
```

---

## Phase 4: MAINTAIN — Keep Index Current

### On File Create
Append new entry to the correct section of INDEX.md. Update DEPENDENCY MAP if other files now import it.

### On File Edit
Re-read the file. Update its entry: purpose, exports, imports. If exports changed, check DEPENDENCY MAP for downstream impact.

### On File Delete
Remove entry from INDEX.md. Check DEPENDENCY MAP — flag any files that imported the deleted file as broken dependencies:
```
⚠️ BROKEN DEPENDENCY: {file} imports from deleted {deleted-file}
```

### On Rename/Move
Update path in entry. Update all DEPENDENCY MAP references.

### Staleness Check
At the start of every session, compare `git diff --name-only` against INDEX.md entries. If mismatches found:
```
⚠️ INDEX STALE: {N} files changed since last scan. Running incremental update...
```

## Integration with Kraken Pipeline

| Kraken Phase | Indexer Action |
|---|---|
| P2: BRAINSTORM | Read INDEX.md for codebase recon. If missing → full scan. |
| P4: PLAN | Reference INDEX.md for file list, dependencies, existing patterns. |
| P6: IMPLEMENT | After EVERY file create/edit/delete → update INDEX.md entry. |
| P8: REVIEW | Cross-reference INDEX.md dependency map for impact analysis. |
| P9: OPTIMIZE | Use INDEX.md to find dead code (files with zero inbound deps). |
| P10: VERIFY | Final INDEX.md update. Verify no broken dependencies. |

---

## Scaling

| Project Size | Scan Strategy |
|---|---|
| <50 files | Full scan every time |
| 50-200 files | Full initial, incremental after |
| 200-1000 files | Full initial, git-diff incremental, skip unchanged |
| >1000 files | Scan by layer/module, parallel processing |

---

## Anti-Patterns

- ❌ Indexing `node_modules` or `dist` — waste of context
- ❌ Including file contents in the index — too large, defeats purpose
- ❌ Manual INDEX.md edits — will be overwritten on next update
- ❌ Skipping index update after file changes — stale index = wrong decisions
- ❌ Indexing .env VALUES — security risk
